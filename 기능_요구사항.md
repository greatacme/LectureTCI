# LectureTCI 기능 요구사항

LectureTCI는 참여형 강의를 위한 웹 화면입니다. 아래 항목에 생각나는 내용을 자유롭게 적어주세요. 비어 있어도 괜찮고, 문장/키워드/예시 화면 흐름 아무 형태나 좋습니다.

## 1\. 프로젝트 개요

* 한 줄 설명: 수강생들이 TCI 검사 예시를 체험하게 하는 웹페이지
* 주 사용자: 강사와 수강생
* 사용할 상황: TCI 검사에 대한 강의
* 가장 중요한 목표: 다수 사용자가 동시에 접속해서 사용
* TCI검사는 MBTI 검사와 유사하다고 생각하자

## 2\. 강의 진행자 기능

* 강의 시작/종료: 수강생용 체험 페이지의 시작과 종료를 제어한다.
* 질문 또는 활동 생성: 
	- 사전에 검사항목 예제를 선정하고 항목별 점수산정 기준을 산정한다.
	- 참여자들이 입력을 마치면 결과를 취합하고 개인별 결과 보기 화면을 오픈한다.
	- 강의를 마치면 참여 페이지를 종료하고 자료를 삭제한다.

* 강의 코드 관리:
	- 체험 시작 버튼을 누르면 해당 강의 세션용 난수 코드가 생성된다.
	- 강의 코드는 예: LT-4821 형식으로 표시한다.
	- 참여자 URL은 매번 새로 생성하지 않고 고정 URL을 사용한다.
	- QR 보기 버튼은 체험 시작 이후 활성화된다.
	- QR 보기 버튼을 누르면 참여자 고정 URL로 이동하는 큰 QR 팝업을 표시한다.
	- 참여자는 QR 또는 고정 URL로 접속한 뒤 강의 코드를 입력해 입장한다.

* 참여자 응답 확인:
	- 참여자 ID와 진행률을 컴팩트한 카드 형태로 확인한다.
	- 상태 텍스트는 표시하지 않고 카드 색상으로 상태를 구분한다.
	- 시작 전/입력 중/입력 완료 상태를 색상으로 구분한다.
	- 100명 정도의 참여자를 고려해 한 화면에 최대한 많은 카드가 보이도록 한다.

* 실시간 피드백:
	- 첫 운영 버전은 2초 polling 기준으로 참여자 진행률을 갱신한다.
	- 향후 필요 시 WebSocket 또는 SSE로 개선한다.
* 결과 저장/내보내기: 수강생이 입력한 내용과 결과는 저장하지 않고, 해당 강의의 체험이 종료되면 폐기된다.

## 3\. 참여자 기능

* 접속 방식:
	- 고정된 참여자 URL로 접속한다.
	- 강사가 화면에 표시하거나 QR로 안내한 URL을 사용한다.
	- 참여자는 강사가 알려준 강의 코드를 입력한 뒤 입장한다.
	- URL 자체에 세션 난수를 넣지 않는다.
* 강의 코드 입력:
	- 강의 코드를 입력해야 검사 화면으로 이동할 수 있다.
	- 프로토타입에서는 코드 검증 없이 화면 흐름만 확인한다.
	- 운영 버전에서는 Spring Boot API로 유효한 세션 코드인지 확인한다.
* 임시 ID 입력:
	- 강의 코드와 함께 임시로 사용할 ID를 입력한다.
	- 이메일이나 전화번호보다는 강사가 배부한 임시 ID를 권장한다.
* 응답 방식: TCI 검사항목별로 응답을 입력한다.
* 익명/실명 여부: 임시 사용할 ID가 배부된다. 개인정보가 포함되는 휴대폰 번호나 메일주소 사용은 가급적 피한다.
* 참여 상태 표시: 참여자 화면에서는 현재 문항 진행률을 표시한다.

## 5\. 화면 구성

* 진행자 화면: 
	- TCI 검사항목 관리 : 데이터베이스로 관리
		- 검사SET 입력/수정/삭제
		- 검사문항 입력/수정/삭제
		- 검사문항은 검사SET별로 별도 관리된다

	- 체험 진행 관리
		- 체험 시작 버튼 : 검사SET를 선택하고 체험 시작을 하면 강의 코드가 생성된다.
		- QR 보기 버튼 : 체험 시작 후 활성화되며, 누르면 참여자 고정 URL의 큰 QR 팝업이 표시된다.
		- 코드 보기 창 : 현재 강의 코드를 크게 표시한다.
		- 참여자 입력 현황 조회 : 참여자의 ID와 진행률을 작은 카드로 표시한다.
		- 상태 구분 : 상태 텍스트 대신 카드 색상으로 시작 전/입력 중/입력 완료를 구분한다.
		- 입력 종료/결과 취합 버튼 : 참여자의 검사항목 입력을 중단시키고 입력된 응답으로 결과를 취합한다.
		- 결과 공유 버튼 : 참여자 화면에 결과보기 버튼을 활성화 시킨다

* 참여자 화면:
	- 강의 코드 입력 창 : 강사가 알려준 강의 코드를 입력한다.
	- ID 입력 창 : ID 입력을 완료하면 검사 화면으로 이동한다.
	- 검사 화면 : 검사항목이 표시되고 응답을 입력할 수 있다. 입력이 종료되면 완료버튼을 누른다.
	- 결과보기 버튼 : 버튼을 누르면 결과 화면으로 이동한다.
	
* 결과 화면:
	- 참여자 화면과 분리된 별도 URL로 접근한다.
	- 검사 결과를 조회할 수 있다.
	
* 결과/대시보드 화면:
	- 프로토타입에서는 개인 결과 화면만 별도 HTML로 분리한다.
	- 운영 버전에서 강사용 전체 결과 대시보드 확장을 검토한다.
* 관리자 또는 설정 화면:
* 모바일 대응 필요 여부: 모바일에 최적화해야 한다.

## 6\. 실시간 기능

* 실시간으로 보여야 하는 정보:
	- 참여자별 입력 진행률
	- 입력 종료 여부
	- 결과 공유 여부
* 새로고침으로 충분한 정보:
	- 검사SET/검사항목 관리 정보
	- 강의 종료 이후의 정리 상태
* 지연 허용 범위: 첫 운영 버전은 2초 정도의 지연 허용
* 동시 접속 예상 인원:100명

## 7\. 데이터와 저장

* 저장해야 할 데이터 : 
	- 검사SET : 검사SET ID/명칭, 일시, 비고
	- 검사항목 : 검사SET ID , 항목 ID/명, 검사문항, 배점, 정렬순서, 비고
	- 강의 세션 : 세션 ID, 검사SET ID, 강의 코드, 세션 상태, 생성 시각, 종료 시각
* 저장하지 않아도 되는 데이터:
	- 강의 종료 후 참여자 ID
	- 강의 종료 후 참여자별 응답 내용
	- 강의 종료 후 개인별 결과
* 강의별 기록 필요 여부:
	- 운영 중에는 강의 세션 단위로 임시 저장한다.
	- 강의 종료 후 참여 데이터와 결과는 삭제한다.
* 개인정보/민감정보 고려사항:
	- 참여자 ID는 개인정보가 아닌 임시 코드 사용을 권장한다.
	- 이메일, 휴대폰 번호 등 식별 가능한 정보 입력은 피한다.
	- Supabase 접속 정보는 브라우저에 노출하지 않는다.

## 11\. 첫 번째 버전 범위

* 시작은 화면 디자인과 버튼 네비게이션만 구현
* 이후 데이터베이스와 상세 기능 구현하자.
* 시험용 데이터는 MBTI 검사를 차용해서 하자
* 첫 프로토타입 화면:
	- 강사 화면 : index.html
	- 참여자 화면 : student.html
	- 결과 화면 : result.html
* 첫 프로토타입에 포함할 주요 흐름:
	- 체험 시작 시 강의 코드 생성
	- QR 보기 팝업
	- 참여자 강의 코드/임시 ID 입력
	- 검사 응답
	- 제출 완료
	- 결과 화면 이동
	- 참여자 입력 현황 컴팩트 카드 표시

## 12\. 메모

* 참여자 URL은 고정하고 강의 코드로 세션을 구분하는 방향을 우선 채택한다.
* QR은 고정 참여자 URL을 안내하고, 강의 코드는 강사 화면에서 별도로 크게 표시한다.
* 고객 검토 중에는 로컬에서 먼저 확인하고, 사용자가 요청할 때만 GitHub Pages에 푸시한다.
